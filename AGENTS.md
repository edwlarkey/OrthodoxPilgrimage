# Orthodox Pilgrimage - Development Guide

## Project Overview

An interactive, community-driven Go web application that maps Orthodox Christian churches across North America containing sacred relics of saints for public veneration. Users browse a modern OpenLayers map to discover holy sites and explore associated saints.

## Tech Stack

- **Backend:** Go 1.24, `net/http` standard library, SQLite (`modernc.org/sqlite`)
- **Database:** SQLite with sqlc for type-safe query generation, embed-based migrations
- **Frontend:** Go `html/template`, HTMX, Leaflet.js, OpenStreetMap
- **Deployment:** Docker (multi-stage), Fly.io
- **Module path:** `github.com/edwlarkey/orthodoxpilgrimage`

## Project Structure

```
cmd/server/          # Application entry point
internal/app/        # Business logic, handlers, seeding, sitemap
internal/app/data/   # Source data (data.json) - single source of truth
internal/db/         # Database connection, migrations
internal/db/migrations/  # SQL migration files (run in sorted order)
internal/db/queries/     # SQL query files consumed by sqlc
internal/db/sqlc/        # Generated Go code from sqlc (do not edit)
internal/ui/         # Template management, embedded assets
```

## Key Conventions

### Agent Mandate: Mandatory Validation

**After making any code changes, agents MUST run `make audit`.** 

This ensures that:
1.  **Code Quality:** `go fmt` and `go vet` are satisfied.
2.  **Linter Compliance:** `golangci-lint` (including `errcheck`, `gosec`, etc.) passes with the project's specific `.golangci.yml` configuration.
3.  **Stability:** All tests, including those with the race detector, pass.
4.  **Data Integrity:** The `data.json` source of truth remains valid.

No change is considered complete until `make audit` has passed successfully.

### Go

- **Module imports:** Use the full module path `github.com/edwlarkey/orthodoxpilgrimage` for all internal imports.
- **Package organization:** Keep `internal/` packages focused. `app` owns HTTP handlers and business logic. `db` owns connection and migrations. `ui` owns template rendering.
- **Error handling:** Use `fmt.Errorf("...: %w", err)` for wrapping errors. Never ignore errors.
- **Context:** Pass `context.Context` as the first argument to all database queries and long-running operations.
- **Interfaces:** Define interfaces at the call site (consumer side) rather than the implementation side. The sqlc `Querier` interface is generated.
- **Naming:** Follow standard Go naming conventions. Abbreviations like `URL`, `ID`, `HTTP` should be all caps.

### Database

- **Migrations:** Add new SQL files to `internal/db/migrations/` with zero-padded prefixes (e.g., `002_add_column.sql`). Migrations are applied in sorted order and tracked in `schema_migrations`.
- **Queries:** Write raw SQL in `internal/db/queries/queries.sql`. Run `make generate` to regenerate Go code.
- **Generated code:** Never edit `internal/db/sqlc/` directly. It is regenerated by sqlc.
- **Null handling:** Use `sql.NullString` for optional columns per sqlc conventions.

### HTTP

- **Routes:** Defined in `Application.Routes()` using `http.NewServeMux`.
- **Handler pattern:** Methods on `*Application` struct. Keep handler logic thin; delegate to `app` package functions.
- **HTMX:** Respond with HTML fragments when `HX-Request` header is present. Set `HX-Push-Url` header for URL state updates.
- **JSON API:** The `/api/v1/churches` endpoint returns JSON with custom marshaled structs. Use `Cache-Control: public, max-age=86400` for caching.

### Templates

- **Embedded:** Templates are embedded via `//go:embed` and loaded by `TemplateManager`.
- **Template structure:** `base.html` is the root template. Page templates extend it via `{{define "content"}}`.
- **Data passing:** Use typed structs (e.g., `ChurchWithRelics`, `SaintWithType`) rather than `map[string]interface{}`.

### Seeding

- **Source of truth:** `internal/app/data/data.json` is the single source of truth for all data.
- **Sync:** Database can be synced from the embedded JSON using the `--seed` flag on startup. This clears and re-inserts all data.
- **Test seeding:** Use `SeedFromReader()` with `strings.NewReader()` or `bytes.NewReader()` for test data.

## Development Commands

```bash

```

## Testing

### Running Tests

```bash
make test
go test -race -v ./...
```

### Writing Tests

- Use `database/sql` with an in-memory SQLite database (`file:memdb?mode=memory&cache=shared`) for all database tests.
- Use `httptest.NewRecorder()` and `httptest.NewRequest()` for HTTP handler tests.
- Use `testify` (`assert` / `require`) for assertions.
- Use `SeedFromReader()` with inline JSON to seed test databases.
- Place tests in the same package as the code they test (e.g., `handler_test.go` in the `app` package).
- Test both success and error paths, including `sql.ErrNoRows` and internal server errors.
- Run with `-race` flag to catch data races.

### Test Patterns

```go
func TestSomething(t *testing.T) {
    // 1. Set up in-memory database
    dsn := "file:memdb?mode=memory&cache=shared"
    dbConn, err := sql.Open("sqlite", dsn)
    require.NoError(t, err)
    defer dbConn.Close()

    // 2. Run migrations
    require.NoError(t, db.MigrateUp(dbConn))

    // 3. Seed test data
    queries := sqlcdb.New(dbConn)
    require.NoError(t, app.SeedFromReader(ctx, queries, strings.NewReader(testDataJSON)))

    // 4. Make request
    req, _ := http.NewRequest("GET", "/path", nil)
    rr := httptest.NewRecorder()
    handler.ServeHTTP(rr, req)

    // 5. Assert
    assert.Equal(t, http.StatusOK, rr.Code)
}
```

## Code Generation

### sqlc

SQL queries live in `internal/db/queries/queries.sql`. Configuration is in `sqlc.yaml`.

```bash
make generate   # Regenerates internal/db/sqlc/
```

After modifying queries or schema, always run `make generate` and commit the generated files.

## Deployment

### Docker

```bash
make build.docker   # Build for all architectures
```

The Dockerfile uses a multi-stage build with `golang:1.26-alpine` and `gcr.io/distroless/static-debian12`.

### Fly.io

Configuration is in `fly.toml`. Deploy via `fly deploy`.

## Adding New Features

1. **New database columns:** Update `001_initial_schema.sql` with a new migration file, then update `queries.sql`.
2. **New queries:** Add to `internal/db/queries/queries.sql`, run `make generate`.
3. **New endpoint:** Add route in `Application.Routes()`, create handler method on `*Application`, write tests.
4. **New template:** Add `.html` file to `internal/ui/templates/`, it will be auto-embedded.
5. **New data:** Update `internal/app/data/data.json`, the next startup will sync it.
6. **Always:** Run `make audit` before committing.
